EPP: Endpoint Protection Platform

Real-time threat prevention before threats even run

💡 What is EPP?

EPP is like a security guard at the door of Sarah's MacBook. It runs continuously in the background and stops known threats (malware, viruses, ransomware) from even executing. If a malicious file tries to run, EPP blocks it immediately—before it can cause harm.

🎬 Scenario: The Suspicious Email

Day 1: Sarah downloads an email attachment that looks like a vendor invoice. Unknown to her, it's actually a trojan malware disguised as a PDF.

What EPP does:

  • ✓ Scans the file before it opens
  • ✓ Matches it against a database of 100+ million known malware signatures
  • ✓ Recognizes the trojan and blocks execution instantly
  • ✓ Quarantines the file (moves it to a safe isolation folder)
  • ✓ Sends an alert to TechCorp's security team

Result: Sarah never even knows the malware existed. EPP stopped it cold.

🔑 Key EPP Capabilities

  • Signature-based detection: Recognizes known threats by their "fingerprint"
  • Heuristic analysis: Spots suspicious behavior patterns (files acting like malware without being a known signature)
  • Real-time scanning: Inspects every file as it runs, before it can infect
  • Quarantine: Safely isolates threats so they can't spread
  • Always-on protection: Runs continuously, even when Sarah isn't using the device
📌 Note for Beginners: EPP is about prevention. It stops threats you know about (signature-based). But what if a brand-new threat emerges that EPP doesn't recognize? That's where EDR comes in.

EDR: Endpoint Detection & Response

Spot unknown threats and respond in real-time

💡 What is EDR?

EDR is like a surveillance camera with a security analyst watching 24/7. It tracks everything happening on Sarah's Mac—every process, file, network connection, and behavior—looking for suspicious activity that might be a threat, even if it's brand new and doesn't match any known malware signature.

🎬 Scenario: The Zero-Day Attack

Day 10: A hacker exploits a brand-new, never-before-seen vulnerability in a popular application on Sarah's Mac. EPP doesn't recognize it because it's zero-day (no signature exists yet).

What EDR does:

  • ✓ Monitors the compromised app's behavior in real-time
  • ✓ Spots suspicious actions: trying to access system files, creating hidden processes, attempting to steal credentials
  • ✓ Triggers an alert: "Suspicious behavior detected on Sarah's Mac"
  • ✓ Records a detailed forensic timeline (what happened, when, in what order)
  • ✓ Alerts TechCorp's SOC (Security Operations Center) team immediately

Response Options:

  • 🔴 Isolate the Mac: Disconnect it from the network (no internet, no internal access) to contain the threat
  • 🔴 Terminate the process: Kill the malicious process automatically
  • 🔴 Rollback: Undo changes the malware made to system files
  • 🔴 Quarantine files: Isolate any files created or modified by the threat

🔑 Key EDR Capabilities

  • Behavioral detection: Spots suspicious behavior, not just known signatures
  • Threat hunting: Analysts can manually search for suspicious patterns
  • Response actions: Isolate devices, kill processes, quarantine files—all automated
  • Forensic timeline: Complete record of what happened (invaluable for investigation)
  • Threat intelligence: Learns from threats across all customer endpoints to improve detection

Deep Visibility

See every process, file, and network activity in real-time

💡 What is Deep Visibility?

Deep Visibility is SentinelOne's forensic engine. It captures and indexes every single event on Sarah's Mac—every process that launched, every file accessed, every network connection, every registry change. You can then search and investigate these events retroactively to answer questions like "What did this malware do?" or "Did the attacker access these files?"

🎬 Scenario: Post-Breach Investigation

Day 15: The SOC team confirms Sarah's Mac was breached 5 days ago. Now they need to investigate: What files did the attacker access? Did they exfiltrate data? What other systems did they touch?

What Deep Visibility does:

  • ✓ Lets the analyst search: "Show me all network connections from Sarah's Mac in the last 7 days"
  • ✓ Displays: 1,247 unique DNS lookups, 89 HTTP/S connections, 3 SSH attempts
  • ✓ Lets them drill down: "Show me files accessed by user 'sarah' on Day 10"
  • ✓ Returns: 456 files accessed; highlights 23 that were read AND modified (suspicious pattern)
  • ✓ Shows the chain of events: Process A launched Process B, which accessed File C, which triggered network connection to IP D

Result: The team reconstructs the attack step-by-step, confirming the attacker accessed customer data files and sent them to an external server at 2:47 AM on Day 10.

🔑 Key Deep Visibility Features

  • Process telemetry: Who launched what process, when, with what arguments
  • File activity: Every file open, read, write, delete, rename, modify
  • Network forensics: DNS queries, IP connections, port usage, data exfiltration patterns
  • Registry/System changes: Configuration modifications, privilege escalations
  • Full chain-of-custody: Parent → child process relationships, file ownership, user context
  • Queryable history: Search by date, user, process name, IP, domain, file path—with AND/OR logic

XDR: Extended Detection & Response

Connect dots across endpoints, networks, cloud, and identity

💡 What is XDR?

XDR is EDR on steroids. It doesn't just monitor Sarah's Mac in isolation. Instead, it correlates data from multiple sources (endpoint, network, cloud, identity, email) to detect sophisticated multi-stage attacks that would be invisible if you only looked at one device. It's like connecting security cameras across an entire building instead of watching one room.

🎬 Scenario: The Multi-Stage APT Attack

Day 20: A nation-state APT (Advanced Persistent Threat) targets TechCorp. The attack uses 4 stages:

  • Stage 1 (Initial Compromise): Phishing email to Sarah (endpoint event)
  • Stage 2 (Lateral Movement): Attacker uses Sarah's credentials to access the network (network event)
  • Stage 3 (Privilege Escalation): Attacker exploits a vulnerability to gain admin rights (endpoint + identity event)
  • Stage 4 (Data Exfiltration): Attacker logs into AWS S3 bucket and downloads customer data (cloud event)

With EPP/EDR alone: Each stage might trigger a separate, isolated alert. Analyst says "Sarah's Mac was compromised" but doesn't connect it to the AWS breach.

With XDR:

  • ✓ SentinelOne correlates all 4 events automatically
  • ✓ Creates a single, unified threat timeline: "APT Campaign detected—4 stages, 8 affected systems"
  • ✓ Immediately shows: Sarah's Mac → Network servers → AWS S3 (data stolen)
  • ✓ Recommends: Isolate Sarah's Mac, reset AWS credentials, investigate 8 systems, alert executive team

Result: Attack detected in minutes (if separate alerts, might take days to correlate manually). Damage minimized.

🔑 Key XDR Capabilities

  • Multi-source correlation: Combines signals from endpoints, network, cloud, identity, email
  • Attack chain detection: Recognizes multi-stage attack patterns automatically
  • Threat scoring: Ranks threats by severity & confidence (not all alerts are equal)
  • Incident grouping: Automatically clusters related events into a single "incident"
  • Cross-platform visibility: Works on Windows, Mac, Linux, and integrates with cloud (AWS, Azure, GCP)

🌍 Real-World XDR Value

In 2023, security teams without XDR averaged 228 days to detect a breach (Verizon DBIR). With XDR, detection time drops to minutes to hours. For TechCorp, that difference means the APT gets caught before exfiltrating 100GB of data instead of after.

STAR Custom Rules

Build your own threat detection logic for your environment

💡 What are STAR Custom Rules?

STAR stands for Singularity Storyline Active Response™. Custom Rules let you write your own detection logic using SentinelOne's query language. Instead of relying only on SentinelOne's pre-built threat detection, you can craft rules tailored to your specific business threats. It's like teaching SentinelOne "Here's what bad looks like in MY company."

🎬 Scenario: TechCorp's Insider Threat Rule

Background: TechCorp discovered that competitors sometimes poach employees to steal source code before they leave. So TechCorp's security team wants to detect suspicious patterns.

The Rule: "Alert if an employee:

  • • Uses an external USB drive for the first time
  • • AND copies >50 files from /home/engineering/ to that USB
  • • AND those files have extensions .py, .java, .cpp (source code)
  • • ALL within a 5-minute window"

Day 25: A disgruntled engineer named Bob plugs in a USB drive and copies 47 Python files to it in 3 minutes. STAR Custom Rule triggers immediately. Alerts Bob's manager, who discovers Bob accepted an offer from TechCorp's competitor. The company secures his machine before he leaves with the code.

🔑 STAR Custom Rule Workflow

  • Identify your threat: What specific attack or suspicious behavior does your company need to detect?
  • Query your data: Use Deep Visibility to find examples of that behavior and craft a query in S1QL 2.0
  • Test the rule: Use the "Simulate Rule" feature to estimate alerts over 7 days based on recent data
  • Deploy: Activate the rule; it now runs continuously in real-time as data streams into Singularity™ Data Lake (in the cloud, not on endpoints)
  • Auto-respond: When the rule triggers, automatically isolate the device, kill the process, or alert the SOC (optional)
  • Tune over time: Investigate real alerts and adjust the rule to reduce false positives
🏗️ Where STAR Rules Execute: STAR rules do NOT run on Sarah's MacBook. Instead, they run in the cloud on the Singularity™ Data Lake. As Deep Visibility events from the agent stream into SDL in real-time, the rules evaluate those events immediately (sub-second latency). This is fundamentally different from endpoint-based rules—the detection happens centrally, not distributed across agents.
🎓 Example STAR Rule (Simplified):
tgt.process.displayName contains "powershell"
AND tgt.process.cmdline contains "System.Net.WebClient"
AND tgt.process.cmdline contains "DownloadString"
→ Auto-isolate endpoint

This rule catches a common malware pattern: PowerShell downloading and executing malicious scripts from the internet. When this query matches an event stream, the rule triggers in SDL and the auto-response (isolate) executes on the endpoint.

Cloud Security: Singularity Cloud (CNAPP)

Unified protection for cloud infrastructure, workloads, and data from build-time to runtime

💡 What is Singularity Cloud?

Singularity Cloud Security is SentinelOne's Cloud-Native Application Protection Platform (CNAPP)—a unified solution that protects cloud infrastructure, workloads, and data across their entire lifecycle. Unlike traditional cloud tools that protect only one layer (e.g., misconfigurations OR runtime threats), Singularity Cloud combines five integrated security functions into one data model.

🔑 The Five Pillars of Singularity Cloud (CNAPP)

  • CSPM (Cloud Security Posture Management): Detects misconfigurations and compliance gaps in your cloud infrastructure (AWS, Azure, GCP). Runs agentless—no software needed.
  • CWPP (Cloud Workload Protection Platform): Real-time runtime protection for VMs, containers, Kubernetes, and serverless workloads. Detects ransomware, zero-days, fileless malware.
  • CIEM (Cloud Infrastructure Entitlement Management): Discovers excessive or misconfigured identity permissions that attackers could exploit for privilege escalation and lateral movement.
  • DSPM (Data Security Posture Management): Identifies and protects sensitive data across cloud storage (S3 buckets, Azure Blob, Google Cloud Storage) from exposure and theft.
  • CDR (Cloud Detection & Response): Real-time threat detection and forensics across cloud infrastructure, correlating events and providing automated response.

🎬 Scenario: The Multi-Layer Cloud Attack

Day 25: An attacker targets TechCorp's AWS infrastructure with a multi-step attack chain:

  • Step 1 (Posture Gap): Attacker discovers an S3 bucket configured with public read access (misconfiguration). No encryption enabled.
  • Step 2 (Entitlement Gap): Attacker gains access to an IAM role with overly permissive permissions (can access any resource in the account).
  • Step 3 (Runtime Attack): Attacker launches a container workload to mine cryptocurrency, consuming 95% CPU and costing TechCorp $50K/month.
  • Step 4 (Data Theft): Attacker accesses the misconfigured S3 bucket and exfiltrates 2GB of customer data.

Traditional Solutions (Point Tools): Three separate teams detect three separate problems at different times:

  • CSPM team: "We found the S3 bucket issue" (but too late—already exploited)
  • CWPP team: "We detected cryptominers" (but no context on how attacker got there)
  • Data team: "We found data exfiltration" (but incident already happened)

With Singularity Cloud (Unified CNAPP):

  • ✓ CSPM detects S3 misconfiguration → alerts in real-time
  • ✓ CIEM identifies overly permissive IAM role → alerts with remediation steps
  • ✓ CWPP detects suspicious container launch → correlates to Step 2 context (attacker assumed to have breached credentials)
  • ✓ DSPM flags unauthorized S3 access → blocks attempt, prevents data theft
  • ✓ CDR builds unified attack timeline: Misconfiguration → Credential Misuse → Workload Compromise → Data Access Blocked
  • ✓ Purple AI generates: "High-confidence APT activity detected. Attack chain: configuration gap → privilege escalation → lateral movement → data access. Recommend: revoke credentials, patch S3, isolate workload."

Result: Attack detected and blocked at Step 4; only misconfiguration exposed, no data stolen.

🏗️ Build-Time vs. Runtime Protection

Build-Time Security (Dev Pipeline):

  • Infrastructure-as-Code (IaC) scanning — detects misconfigurations in Terraform, CloudFormation before deployment
  • Secrets scanning — finds 750+ types of hardcoded secrets (API keys, passwords, tokens) in repositories
  • Container scanning — identifies vulnerable images, malware before they ship to production
  • Shift-left security — catch misconfigurations in dev, not production

Runtime Security (Production):

  • Real-time malware/ransomware detection in running containers and VMs
  • Container drift detection — alerts if containers are modified after launch (attackers install backdoors)
  • Process execution analysis — detects suspicious commands, privilege escalation attempts
  • Network forensics — tracks data exfiltration, lateral movement attempts
  • Zero-day protection via behavioral AI (not just signatures)

Attackers chain exposures across all five layers in a single attack. A misconfiguration becomes lateral movement; a credential becomes privilege escalation. Point tools (e.g., CSPM-only) see only one layer. A unified CNAPP sees the attack chain and blocks it.

SentinelOne combines agentless capabilities (CSPM, KSPM, vulnerability management) with strong agent-based runtime protection (CWPP) — the best of both worlds.

Identity Security (Singularity for Identity)

Protect user credentials and access rights

💡 What is Identity Security?

Identity is often the weakest link in security. Once an attacker steals Sarah's credentials (username & password), they can impersonate her, access her files, and move laterally through the network. Singularity for Identity protects credentials and detects abuse:

  • Detects credential theft and compromised credentials
  • Monitors for unusual login patterns (Sarah logging in from 3 countries in 30 minutes = compromise)
  • Manages Active Directory (AD) security to prevent lateral movement
  • Integrates with SIEM/XDR to correlate identity threats with endpoint threats

🎬 Scenario: Credential Theft & Lateral Movement

Day 22: An attacker who breached Sarah's Mac steals her Windows domain credentials. They then use Sarah's identity to access the Finance team's shared folder and the HR database.

Without Identity Security: Security team doesn't know Sarah's credentials were compromised. Attacker quietly reads sensitive data for days.

With Singularity for Identity:

  • ✓ Detects credential theft event on Sarah's Mac (EDR alert)
  • ✓ Monitors Sarah's identity: flagged login from new IP, unusual time-of-day, access to systems she never uses
  • ✓ Automatic response: Force Sarah to re-authenticate, revoke stolen credentials, alert her manager
  • ✓ Blocks the attacker's lateral movement (even though they have Sarah's password, security policies prevent access)

Result: Attacker's access revoked immediately. No sensitive data stolen because identity protection caught the abuse.

🔑 Identity Security Capabilities

  • Credential Monitoring: Detects stolen passwords, API keys, tokens
  • Anomalous Login Detection: Flags impossible travel, unusual access times, new devices
  • Privilege Escalation Blocking: Prevents lateral movement even with valid credentials
  • Active Directory Hardening: Protects AD from attacks like Kerberoasting, Pass-the-Hash
  • Conditional Access: Requires additional authentication for sensitive systems (e.g., MFA for finance access)
  • XDR Integration: Links identity threats to endpoint threats for full-picture visibility
